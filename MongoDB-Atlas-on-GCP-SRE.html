MongoDB Atlas on GCP - SRE Perspective

This overview analyzes MongoDB Atlas on Google Cloud Platform (GCP) through an SRE lens, focusing on architecture, reliability, observability, and infrastructure-as-code.

1. Architectural & Network Topology

From an infrastructure standpoint, Atlas is a fully managed service, but it resides in a "tenant" GCP VPC that is peered or connected to your "consumer" VPC.

  • Cluster Composition:

    • Replica Sets: The default deployment is a 3-node replica set spread across 3 separate GCP Zones within a single Region for high availability (HA).
    • Sharded Clusters: For horizontal scaling, Atlas automates sharding (mongos routers + config servers + shard replica sets).
    • Global Clusters: You can deploy nodes across multiple GCP regions (e.g., us-central1, europe-west1) for low-latency local reads or cross-region DR.

  • Network Connectivity (Critical for SREs):

    • Private Service Connect (PSC): (Recommended) The modern standard for GCP. It allows unidirectional private connectivity from your VPC to Atlas without the complexity of transitivity issues associated with VPC Peering. It exposes Atlas via a private IP in your subnet.
    • VPC Peering: The legacy method. It connects the Atlas VPC network directly to yours. Warning: GCP VPC peering is non-transitive. If you peer App VPC ↔ Atlas VPC, you cannot hop from VPN → App VPC → Atlas VPC without a proxy.
    • IP Access Lists: Functions as a network firewall. Even with peering/PSC, you must whitelist CIDR blocks or specific IPs.

2. Reliability & Disaster Recovery (DR)

Atlas handles the heavy lifting of database durability, but SREs own the configuration of that durability.

  • SLA: 99.995% uptime guarantee for multi-zone clusters.
  • Failover: Automated election and failover (typically <10 seconds). Drivers automatically handle the topology change if the connection string (SRV record) is used correctly.
  • Backup Strategy:
    • Continuous Cloud Backups: Uses GCP native snapshots. You can enable Point-In-Time Recovery (PITR) to restore to a specific second (usually roughly a 24-hour window).
    • Cross-Region Snapshots: You can configure backup policies to copy snapshots to a different GCP region (e.g., snapshot in Iowa, copy to Belgium) for region-level DR.
  • Testing DR: You can spin up a "clone" of your cluster from a backup into a lower environment (e.g., Staging) to verify data integrity and recovery scripts without impacting production.

3. Observability & Monitoring

This is often the biggest friction point for GCP SREs because Atlas does not "natively" push metrics to Google Cloud Monitoring (formerly Stackdriver) out of the box without configuration.

  • Native Tools: Atlas UI provides a "Real-Time Performance Panel" (live ops) and "Query Profiler" (identifying slow queries/missing indexes).
  • Google Cloud Monitoring (Stackdriver) Integration:
    • Pull Model (Prometheus): The standard SRE pattern is to use the Prometheus Integration. Atlas exposes a metrics endpoint. You can run a mongodb_exporter or configure Google Cloud Managed Service for Prometheus to scrape this endpoint.
    • Push Model (Custom): Alternatively, you can write a small serverless function (Cloud Run/Functions) that polls the Atlas Administration API and pushes custom metrics to Cloud Monitoring, but this adds maintenance overhead.
  • Logs: You can push MongoDB logs (audit, error, etc.) to a Google Cloud Storage bucket or directly to Cloud Logging via an export configuration.

4. Security & Compliance

  • Encryption:
    • In-Transit: TLS 1.2+ is mandatory.
    • At-Rest: Encrypted by default using Google's internal keys.
    • Customer Managed Keys (CMK): You can use Google Cloud KMS to hold the master encryption key (MEK) if you have strict compliance requirements (e.g., HIPAA, PCI).
  • Authentication:
    • IAM: Supports "Database Users" (built-in) or peering with LDAP/OIDC.
    • GCP Service Accounts: You can federate GCP Service Account identity to manage Atlas resources (via the Atlas API), reducing the need for long-lived API keys.

5. Infrastructure as Code (Terraform)

Managing Atlas via ClickOps (UI) is an anti-pattern for SREs. Use the official Terraform Provider (mongodb/mongodbatlas).

Key Resources:

  • mongodbatlas_cluster: Defines instance size (e.g., M10, M30), region, and disk auto-scaling settings.
  • mongodbatlas_privatelink_endpoint: Manages the GCP Private Service Connect setup.
  • mongodbatlas_database_user: Manages DB credentials programmatically (can rotate these via Vault or Secret Manager).

Example Snippet (PSC Setup):

resource "mongodbatlas_privatelink_endpoint" "atlas_psc" {
  project_id    = var.atlas_project_id
  provider_name = "GCP"
  region        = "us-central1"
}

# Note: You then have to create a Google Compute Forwarding Rule
# pointing to the service attachment provided by the resource above.

6. SRE "Gotchas" & Best Practices

Area Best Practice
Cost Enable Auto-Scaling for storage (disk grows automatically) but be careful with downscaling (often manual). Monitor Disk IOPS as higher tiers (M30+) act differently than lower tiers.
Connection Always use the SRV Connection String (mongodb+srv://...). This abstracts the node IPs, allowing Atlas to replace/rotate nodes without breaking app config.
Bursting Standard GCP persistent disks have I/O limits. If you anticipate spiky traffic, ensure your chosen instance size (M-tier) supports the required IOPS, or you will see high IOWait.
Version Set the mongo_db_major_version in Terraform. Do not let it "float" to avoid unexpected major version upgrades during maintenance windows.
